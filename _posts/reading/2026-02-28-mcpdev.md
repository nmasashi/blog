## 第1章MCPサーバー開発の基礎知識

### MCPとは何か

Model Context Protocol (MCP) とは: LLMと外部システムを接続するために標準化されたプロトコル

なぜMCPが必要なのか: LLMの適用範囲が広がったことで、再利用可能なコンポーネントへの需要が高まったため

### MCPの3つの主要コンポーネント

- MCPクライアント
  - ユーザーとMCPサーバーを仲介する
  - MCPサーバーをホストする
  - Claude for Desctop, Continueなどが実例
  - 主な機能
    - MCPサーバープロセスの起動と管理
    - ユーザーインターフェースの提供
    - LLMとMCPサーバーの仲介

- MCPサーバー
  - 特定の機能やデータソースへのアクセスを提供する専門的なコンポーネント
  - 役割
    - 特定のツール（機能）の実装と公開
    - 外部APIやデータベースとの連携
    - データ取得と変換処理
    - エラーハンドリングと適切なレスポンスの生成

- LLM
  - MCPエコシステムの頭脳
  - ユーザーの自然言語から要求を理解して適切なMCPツールを選択して、実行結果を解釈してユーザーに返答する
  - MCPサーバーと直接やりとりするわけではなく、MCPクライアントを通してやり取りする

### 開発環境の構成

書籍で紹介されている内容が少しだけ古いみたい

```sh
# これがあれば
uv add mcp

# 多分こいつらはいらない
uv add "mcp[server]"
uv add "mcp[all]"
```

`mcp.server.Server`を使用しているけど低レベルAPIらしい。サンプルにある`tool`デコレータもない。
推奨は`FastMCP`を使うみたい。

1.3.2節の最新版コード

```python
import asyncio
from mcp.server.fastmcp import FastMCP


async def test_installation():
    print("MCPインストールテスト開始")

    try:
        server = FastMCP("test-server")
        print("サーバー作成完了")
    except Exception as e:
        print(f"サーバー作成失敗: {e}")

    @server.tool()
    async def test_tool() -> str:
        return "Hello MCP"

    print("ツール登録成功")
    print("\nインストール完了! MCPサーバー開発の準備ができました")


if __name__ == "__main__":
    asyncio.run(test_installation())
```

## 第2章 MCPアーキテクチャの理解

### MCPプロトコルの全体像

#### クライアント・サーバーモデルの詳細

MCPは古典的なクライアント・サーバーモデルを採用している

一般的なwebアプリのようなリクエスト・レスポンス型の接続ではなく、
クライアントがサーバープロセスを起動して永続的な双方向チャンネルを確立する点が特徴。

このような永続接続モデルを利用することで、HTTP接続のように毎回接続する必要がなくなり、
接続のオーバーヘッドが削減されて低レイテンシでの通信が可能になる。
また、サーバー側での状態管理が容易になり、より洗練された機能の提供も容易になる。

非同期処理にも対応しだした

#### JSON-RPC ベースの通信仕様

MCP の通信プロトコルはJSON-RPC 2.0 をベースにしている

JSON-RPC 2.0 には決まったフィールドがある

リクエストに含まれるフィールド

| フィールド | 意味                            |
| ---------- | ------------------------------- |
| jsonrpc    | プロトコルバージョン（常に2.0） |
| method     | 呼び出すメソッド名              |
| params     | メソッドに渡すパラメータ        |
| id         | リクエストを識別する一意のID    |

#### stdioとSSEの使い分け

MCPは下記の二つのトランスポート層をサポートしている

- stdio（標準入出力）トランスポート
  - デフォルトかつ一般的な通信方式
  - 親プロセス（クライアント）と子プロセス（サーバー）の間で標準入力と標準出力を使用してメッセージ交換
  - ローカルマシン上でのみ動作し、リモートサーバーとの通信は負荷
  - 1つのクライアントに対して、1つのサーバープロセスが必要にある（複数のクライアントを使用する場合、リソース効率は良くない）

- SSE (Server-Sent Events) トランスポート
  - HTTP上で動作する単方向のストリーミングプロトコル
  - MCPでは双方向通信を実現するため、SSEとHTTP POSTを組み合わせている
  - リモートサーバーとの通信が可能
  - 1つのサーバーインスタンスで複数のクライアントが処理できてリソース効率がよい

デスクトップアプリを開発する場合はstdioを選択し、webサーバーなどクラウドを使用する場合はSSEを選択する

### MCPサーバーの設計パターン

- シンプルラッパーパターン
  - 既存のAPIサービスをLLMが理解できる形式に変換する薄い層を提供する
  - 外部のWeb APIが返すJSON形式のデータを受け取り、それをLLMが理解しやすい形式に変換して返す
  - 既存のREST APIをLLMから使用したい場合に良い

- カスケード処理パターン
  - 複数のAPIを呼び出す
  - 一連のAPIを統合して呼び出して返答する
  - エラー処理はグレースフルデグラデーション（優雅な機能縮退）を採用
    - グレースフルデグラデーション：一部のAPIが失敗しても成功したもので返答する

- ナレッジベースパターン
  - MCPサーバー自身が管理する文書やデータベースから情報を取得

- ステートレスパターン
  - 各リクエストを独立して処理し、リクエスト間で状態を保持しない
  - 水平スケーリングがしやすい
  - サーバーが落ちても別インスタンスが立ち上がっていたらOK
  - 会話の文脈の保持ができないため複数回のやり取りにまたがる複雑な処理に向かない

- ステートフルパターン
  - 接続期間中、クライアント固有の状態を保持
  - ユーザーの過去の質問を記憶し、文脈を考慮した応答を生成可能
  - 複雑な分析処理を段階的に実行し、最終的な結果を構築可能
  - スケールアウトが困難

ステートレスとステートフルを組み合わせたパターンもある。
基本的にはステートレスで稼働して必要に応じて外部DBに状態を保存するなど。

### MCPサーバーのスケーラビリティ

セマンティックキャッシュ：同じ質問に対して同じ回答をすることを目的としたキャッシュ

### セキュリティアーキテクチャ

レート制限

- 1分間100リクエスト
- 1日10,000トークンまで

ログ

- パスワードやAPIキーがログに残らないようにする
- 個人情報などのPIIは特別な理由がない限り残さない
- プライバシーに関する情報はマスキングするなどの考慮が必要

### 他プロトコルアーキテクチャとの比較

|                                  | MCP                                          | REST                                                               |
| -------------------------------- | -------------------------------------------- | ------------------------------------------------------------------ |
| パラメータの命名やレスポンス構造 | LLMの理解しやすさを優先                      | 人間の理解しやすさを優先                                           |
| 設計の考え方                     | 「ユーザ」や「商品」などのリソースを操作する | 「天気を取得する」や「ドキュメントを取得する」などの機能を操作する |
| 接続モデル                       | 短命な要求の繰り返し                         | 永続的な接続を維持することもある                                   |

プロバイダ特有の機能

- Function Calling: OpneAIのモデルで動作する
- Tool Use: Claude系のモデルで機能する

高頻度で使用される基本的な機能はプロバイダー固有の機能で実装し、特殊なドメイン知識や組織固有の機能はMCPサーバーとして実装するハイブリットアプローチが有効

MCPが不向きなもの

- 高頻度、低レイテンシが要求される場面
- 大量データの転送
- 厳密なトランザクション
- レガシーシステムとの統合
- プライバシーが極めて重要な場面
