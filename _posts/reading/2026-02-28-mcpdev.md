## 第1章MCPサーバー開発の基礎知識

### MCPとは何か

Model Context Protocol (MCP) とは: LLMと外部システムを接続するために標準化されたプロトコル

なぜMCPが必要なのか: LLMの適用範囲が広がったことで、再利用可能なコンポーネントへの需要が高まったため

### MCPの3つの主要コンポーネント

- MCPクライアント
  - ユーザーとMCPサーバーを仲介する
  - MCPサーバーをホストする
  - Claude for Desctop, Continueなどが実例
  - 主な機能
    - MCPサーバープロセスの起動と管理
    - ユーザーインターフェースの提供
    - LLMとMCPサーバーの仲介

- MCPサーバー
  - 特定の機能やデータソースへのアクセスを提供する専門的なコンポーネント
  - 役割
    - 特定のツール（機能）の実装と公開
    - 外部APIやデータベースとの連携
    - データ取得と変換処理
    - エラーハンドリングと適切なレスポンスの生成

- LLM
  - MCPエコシステムの頭脳
  - ユーザーの自然言語から要求を理解して適切なMCPツールを選択して、実行結果を解釈してユーザーに返答する
  - MCPサーバーと直接やりとりするわけではなく、MCPクライアントを通してやり取りする

### 開発環境の構成

書籍で紹介されている内容が少しだけ古いみたい

```sh
# これがあれば
uv add mcp

# 多分こいつらはいらない
uv add "mcp[server]"
uv add "mcp[all]"
```

`mcp.server.Server`を使用しているけど低レベルAPIらしい。サンプルにある`tool`デコレータもない。
推奨は`FastMCP`を使うみたい。

1.3.2節の最新版コード

```python
import asyncio
from mcp.server.fastmcp import FastMCP


async def test_installation():
    print("MCPインストールテスト開始")

    try:
        server = FastMCP("test-server")
        print("サーバー作成完了")
    except Exception as e:
        print(f"サーバー作成失敗: {e}")

    @server.tool()
    async def test_tool() -> str:
        return "Hello MCP"

    print("ツール登録成功")
    print("\nインストール完了! MCPサーバー開発の準備ができました")


if __name__ == "__main__":
    asyncio.run(test_installation())
```

## 第2章 MCPアーキテクチャの理解

### MCPプロトコルの全体像

#### クライアント・サーバーモデルの詳細

MCPは古典的なクライアント・サーバーモデルを採用している

一般的なwebアプリのようなリクエスト・レスポンス型の接続ではなく、
クライアントがサーバープロセスを起動して永続的な双方向チャンネルを確立する点が特徴。

このような永続接続モデルを利用することで、HTTP接続のように毎回接続する必要がなくなり、
接続のオーバーヘッドが削減されて低レイテンシでの通信が可能になる。
また、サーバー側での状態管理が容易になり、より洗練された機能の提供も容易になる。

非同期処理にも対応しだした

#### JSON-RPC ベースの通信仕様

MCP の通信プロトコルはJSON-RPC 2.0 をベースにしている

JSON-RPC 2.0 には決まったフィールドがある

リクエストに含まれるフィールド

| フィールド | 意味                            |
| ---------- | ------------------------------- |
| jsonrpc    | プロトコルバージョン（常に2.0） |
| method     | 呼び出すメソッド名              |
| params     | メソッドに渡すパラメータ        |
| id         | リクエストを識別する一意のID    |

#### stdioとSSEの使い分け

MCPは下記の二つのトランスポート層をサポートしている

- stdio（標準入出力）トランスポート
  - デフォルトかつ一般的な通信方式
  - 親プロセス（クライアント）と子プロセス（サーバー）の間で標準入力と標準出力を使用してメッセージ交換
  - ローカルマシン上でのみ動作し、リモートサーバーとの通信は負荷
  - 1つのクライアントに対して、1つのサーバープロセスが必要にある（複数のクライアントを使用する場合、リソース効率は良くない）

- SSE (Server-Sent Events) トランスポート
  - HTTP上で動作する単方向のストリーミングプロトコル
  - MCPでは双方向通信を実現するため、SSEとHTTP POSTを組み合わせている
  - リモートサーバーとの通信が可能
  - 1つのサーバーインスタンスで複数のクライアントが処理できてリソース効率がよい

デスクトップアプリを開発する場合はstdioを選択し、webサーバーなどクラウドを使用する場合はSSEを選択する

### MCPサーバーの設計パターン

- シンプルラッパーパターン
  - 既存のAPIサービスをLLMが理解できる形式に変換する薄い層を提供する
  - 外部のWeb APIが返すJSON形式のデータを受け取り、それをLLMが理解しやすい形式に変換して返す
  - 既存のREST APIをLLMから使用したい場合に良い

- カスケード処理パターン
  - 複数のAPIを呼び出す
  - 一連のAPIを統合して呼び出して返答する
  - エラー処理はグレースフルデグラデーション（優雅な機能縮退）を採用
    - グレースフルデグラデーション：一部のAPIが失敗しても成功したもので返答する

- ナレッジベースパターン
  - MCPサーバー自身が管理する文書やデータベースから情報を取得

- ステートレスパターン
  - 各リクエストを独立して処理し、リクエスト間で状態を保持しない
  - 水平スケーリングがしやすい
  - サーバーが落ちても別インスタンスが立ち上がっていたらOK
  - 会話の文脈の保持ができないため複数回のやり取りにまたがる複雑な処理に向かない

- ステートフルパターン
  - 接続期間中、クライアント固有の状態を保持
  - ユーザーの過去の質問を記憶し、文脈を考慮した応答を生成可能
  - 複雑な分析処理を段階的に実行し、最終的な結果を構築可能
  - スケールアウトが困難

ステートレスとステートフルを組み合わせたパターンもある。
基本的にはステートレスで稼働して必要に応じて外部DBに状態を保存するなど。

### MCPサーバーのスケーラビリティ

セマンティックキャッシュ：同じ質問に対して同じ回答をすることを目的としたキャッシュ

### セキュリティアーキテクチャ

レート制限

- 1分間100リクエスト
- 1日10,000トークンまで

ログ

- パスワードやAPIキーがログに残らないようにする
- 個人情報などのPIIは特別な理由がない限り残さない
- プライバシーに関する情報はマスキングするなどの考慮が必要

### 他プロトコルアーキテクチャとの比較

|                                  | MCP                                          | REST                                                               |
| -------------------------------- | -------------------------------------------- | ------------------------------------------------------------------ |
| パラメータの命名やレスポンス構造 | LLMの理解しやすさを優先                      | 人間の理解しやすさを優先                                           |
| 設計の考え方                     | 「ユーザ」や「商品」などのリソースを操作する | 「天気を取得する」や「ドキュメントを取得する」などの機能を操作する |
| 接続モデル                       | 短命な要求の繰り返し                         | 永続的な接続を維持することもある                                   |

プロバイダ特有の機能

- Function Calling: OpneAIのモデルで動作する
- Tool Use: Claude系のモデルで機能する

高頻度で使用される基本的な機能はプロバイダー固有の機能で実装し、特殊なドメイン知識や組織固有の機能はMCPサーバーとして実装するハイブリットアプローチが有効

MCPが不向きなもの

- 高頻度、低レイテンシが要求される場面
- 大量データの転送
- 厳密なトランザクション
- レガシーシステムとの統合
- プライバシーが極めて重要な場面

```python
@mcp.tool()
async def get_forecast(latitude: float, longitude: float) -> str:
    """Get weather forecast for a location.

    Args:
        latitude: Latitude of the location
        longitude: Longitude of the location
    """
    # First get the forecast grid endpoint
    points_url = f"{NWS_API_BASE}/points/{latitude},{longitude}"
    points_data = await make_nws_request(points_url)

    if not points_data:
        return "Unable to fetch forecast data for this location."

    # Get the forecast URL from the points response
    forecast_url = points_data["properties"]["forecast"]
    forecast_data = await make_nws_request(forecast_url)

    if not forecast_data:
        return "Unable to fetch detailed forecast."

    # Format the periods into a readable forecast
    periods = forecast_data["properties"]["periods"]
    forecasts = []
    for period in periods[:5]:  # Only show next 5 periods
        forecast = f"""
{period["name"]}:
Temperature: {period["temperature"]}°{period["temperatureUnit"]}
Wind: {period["windSpeed"]} {period["windDirection"]}
Forecast: {period["detailedForecast"]}
"""
        forecasts.append(forecast)

    return "\n---\n".join(forecasts)

```

MCPサーバーでのエラーハンドリング

- エラーは文字列として返却する
- 次のアクションの指針を含める
- 部分的な成功を正しく扱う

データ成形

- 重要な情報を上位に配置
- 欠損値に対してデフォルト値を設定しておく
- 一貫性のあるフォーマットを使用する

 実装パターンの抽出

- ツール定義パターン
  - docstringや引数名などでLLMが関数の内容を理解できるようにする
- データ変換パターン
  - APIが複雑な入れ子構造のデータだったとしてもそれをLLMが読みやすい形式に変換する
- 非同期処理パターン
- エラーハンドリングパターン
  - 例外のスタックトレースを返すのではなく、LLMが理解できるような形式にして返す

### 天気予報MCPサーバーの構築

サンプルコード

```python
import os
import httpx
from typing import Any, Optional
from mcp.server.fastmcp import FastMCP
from dotenv import load_dotenv

# .envファイルから環境変数を読み込む
load_dotenv()

# 環境変数から設定を取得
API_KEY = os.getenv("OPENWEATHER_API_KEY")
BASE_URL = os.getenv("OPENWEATHER_BASE_URL", "https://api.openweathermap.org/data/2.5")

# APIキーが設定されているかチェック
if not API_KEY:
    raise ValueError(
        "OPENWEATHER_API_KEY is not set. Please create a .env file with your API key."
    )

# MCPサーバーの初期化
mcp = FastMCP("openweathermap-service")

print("Weather MCP Server initialized")
print(f"Using API base URL: {BASE_URL}")


async def fetch_weather_data(city: str) -> Optional[dict[str, Any]]:
    """
    Open Weather Map APIから天気データを取得する

    Args:
        city: 都市名（英語）

    Returns:
        天気データのdict、エラーの場合はNone
    """
    url = f"{BASE_URL}/weather"
    params = {
        "q": city,
        "appid": API_KEY,
        "units": "metric",  # 摂氏で温度を取得
        "lang": "ja",  # 日本語で天気の説明を取得
    }

    async with httpx.AsyncClient() as client:
        try:
            response = await client.get(url, params=params, timeout=10.0)
            response.raise_for_status()
            return response.json()
        except httpx.HTTPStatusError as e:
            if e.response.status_code == 404:
                print(f"City not found: {city}")
            elif e.response.status_code == 401:
                print("Invalid API key")
            else:
                print(f"HTTP error: {e}")
            return None
        except Exception as e:
            print(f"Error fetching weather data: {e}")
            return None


def format_weather_response(data: dict[str, Any]) -> str:
    """
    天気データを読みやすい形式にフォーマットする

    Args:
        data: Open Weather Map APIのレスポンス

    Returns:
        フォーマットされた天気情報
    """
    # データから必要な情報を抽出
    city_name = data.get("name", "不明")
    country = data.get("sys", {}).get("country", "")

    # 天気情報
    weather = data.get("weather", [{}])[0]
    description = weather.get("description", "不明")

    # 温度情報（すでに摂氏）
    main = data.get("main", {})
    temp = main.get("temp", 0)
    feels_like = main.get("feels_like", 0)
    temp_min = main.get("temp_min", 0)
    temp_max = main.get("temp_max", 0)
    humidity = main.get("humidity", 0)
    pressure = main.get("pressure", 0)

    # 風情報
    wind = data.get("wind", {})
    wind_speed = wind.get("speed", 0)
    wind_deg = wind.get("deg", 0)

    # 風向きを方角に変換
    wind_direction = convert_degrees_to_direction(wind_deg)

    # フォーマットされた応答を作成
    response = f"""
{city_name}, {country} の現在の天気

天候: {description}
現在の気温: {temp:.1f}°C
体感温度: {feels_like:.1f}°C
最低/最高気温: {temp_min:.1f}°C / {temp_max:.1f}°C
湿度: {humidity}%
気圧: {pressure} hPa
風: {wind_direction} {wind_speed} m/s
"""

    return response.strip()


def convert_degrees_to_direction(degrees: float) -> str:
    """風向き（度）を方角に変換する"""
    directions = [
        "北",
        "北北東",
        "北東",
        "東北東",
        "東",
        "東南東",
        "南東",
        "南南東",
        "南",
        "南南西",
        "南西",
        "西南西",
        "西",
        "西北西",
        "北西",
        "北北西",
    ]
    index = round(degrees / 22.5) % 16
    return directions[index]


@mcp.tool()
async def get_weather(city: str) -> str:
    """
    指定された都市の現在の天気を取得します

    Args:
        city: 都市名（例: Tokyo, London, New York）

    Returns:
        天気情報のフォーマットされた文字列
    """
    # 入力の検証
    if not city or not city.strip():
        return "エラー: 都市名を入力してください。"

    # 天気データを取得
    weather_data = await fetch_weather_data(city.strip())

    if weather_data is None:
        return f"すみません。'{city}' の天気情報を取得できませんでした。都市名を確認してもう一度お試しください。"

    # フォーマットして返す
    return format_weather_response(weather_data)


if __name__ == "__main__":
    # サーバーを起動
    mcp.run(transport="stdio")
```
